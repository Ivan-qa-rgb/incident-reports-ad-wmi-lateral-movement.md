# Red Team чек-лист: компрометация AD-учётки и WMI lateral movement

**Сценарий:** доступ к доменной учётке → латеральное перемещение через WMI/PowerShell на контроллер домена и критичные сервера.  
**Цель:** показать типичные техники и команды атакующего (Red) и связать их с тем, что должен видеть защитник (Blue).

---

## 1. Initial Access: как добывают учётку

Типовые варианты (в легальной лаборатории / пентесте):

### Фишинг / delivery

- Отправка письма с вложением/ссылкой на payload (macro, HTA, JS, ISO/VHD).
- Использование легитимных каналов (O365, корпоративная почта) в рамках согласованного сценария.

### Credential harvesting

- Захват хеша/пароля через NTLM relay / HTTP/SMB poisoning.
- Локальный credential dumping (LSASS, SAM, DPAPI) после получения foothold.

### Password spraying / ограниченный bruteforce

- Password spraying по спискам пользователей (SAM / UPN).
- Попытки входа по VPN/RDP с базовыми шаблонами паролей (если это разрешено правилами пентеста).

### Reuse

- Проверка повторного использования паролей на разных сервисах/доменах.
- Тестирование найденных паролей из прошлых утечек.

**Как это выглядит для Blue**

- Серия 4625 → первый 4624 по одной учётке.
- Нестандартные источники логинов (новые хосты, IP, гео).
- Необычные VPN/RDP‑подключения.

## 2. Фиксация foothold и подготовка к lateral movement

Когда Red уже получил валидную доменную учётку и начальный доступ.

### Проверка контекста

- Определить, к какой группе относится учётка (Domain Users, локальный админ, сервисный аккаунт).
- Собрать базовую информацию об окружении: домен, контроллеры, основные сегменты и сервисы.

### Минимальный recon

Классика:

- `net user`
- `net group /domain`

PowerShell / .NET (в согласованном пентесте):

- `Get-ADUser`
- `Get-ADComputer`

Сбор информации о доступных ресурсах:

- файловые шары;
- серверы приложений;
- критичные сервисы.

**Что видит Blue**

- Появление логинов с необычных хостов.
- Активность учётки на нетипичных ресурсах.
- Первые команды PowerShell от этой учётки/хоста.

## 3. Использование PowerShell для pivot’а

PowerShell — основной «многофункциональный нож» Red Team.

### Интерактивный PowerShell

Запуск:

- `powershell.exe -NoProfile -ExecutionPolicy Bypass -EncodedCommand <...>`

Обфускация:

- base64‑encoded команды;
- `Invoke-Expression`, стеганография в строках и т.д.

### PowerShell Remoting

Включение / использование WinRM:

- `Enable-PSRemoting` (если разрешено сценарием).
- `Enter-PSSession`, `Invoke-Command` для удалённого исполнения.

### Модули для AD/сетей

- Импорт модулей, работающих с AD / сетевой инфраструктурой (в рамках белого сценария).

**Что видит Blue**

События:

- 4104 (PowerShell Script Block Logging).
- 4688 (создание процессов).
- Sysmon‑события по запуску PowerShell.

Необычные параметры:

- `-EncodedCommand`.
- `-ExecutionPolicy Bypass`.
- Обфусцированный код.

## 4. WMI lateral movement

Классика Red Team для тихого lateral movement.

### Использование WMI для удалённого исполнения

Через `wmic`:

- `wmic /node:<target> process call create "cmd.exe /c <команда>"`

Через PowerShell:

- `Invoke-WmiMethod` с указанием удалённого хоста.
- `Get-WmiObject` / `Get-CimInstance` + дальнейшее выполнение команд.

### Цель

- Выполнить payload/скрипт на удалённом хосте без явного RDP:
  - деплой агента/бекдора;
  - выполнение команд для расширения foothold.

### Частые условия

- Учётка с правами локального администратора на целевом хосте.
- Отключённое или плохо настроенное журналирование PowerShell/WMI.

**Что видит Blue**

- Входы учётки на новые хосты.
- WMI‑события и появление неожиданных процессов.
- PowerShell‑активность, связанная с WMI.

## 5. Закрепление (persistence) для Red Team

После успешного lateral movement задача Red — закрепиться.

### Планировщик задач (Task Scheduler)

- Создание задач, запускающих PowerShell/EXE:
  - «на логон»;
  - «на старте системы»;
  - «по расписанию».
- Маскировка под легитимные названия.

### Автозагрузка в реестре

Ключи:

- `HKLM\Software\Microsoft\Windows\CurrentVersion\Run(*)`
- `HKCU\Software\Microsoft\Windows\CurrentVersion\Run(*)`

Использование нестандартных веток и ключей для усложнения детекта.

### Сервисы

Регистрация собственного сервиса:

- `sc.exe create ...`
- Через PowerShell (`New-Service` и т.п.).

### WMI‑подписки

Создание WMI Event Subscription:

- триггер по логону/событию/таймеру;
- выполнение PowerShell/EXE при наступлении события.

**Что делает Blue**

- Целенаправленно ищет:
  - задачи планировщика;
  - ключи Run/Services/IFEO;
  - WMI‑подписки;
  - нестандартные сервисы.
- Сопоставляет это с timeline и IOC.

## 6. Масштабирование и скрытность (оптимизация под «тихий» пентест)

Red‑подход к усложнению работы Blue.

### Минимизировать шум

- Ограничивать количество попыток логина (избегать шумного bruteforce).
- Использовать уже существующие каналы:
  - корпоративный VPN;
  - jump‑host’ы;
  - админские рабочие станции (где это возможно по сценарию).

### Использовать «живущие» в системе инструменты

- WMI, PowerShell, `net`, `schtasks`, `sc`, встроенные библиотеки .NET.
- По максимуму избегать загрузки сторонних EXE:
  - меньше шансов быть пойманными AV/EDR по сигнатурам.

### Маскировка

- Легитимные имена задач, служб, процессов.
- Встраивание под уже существующие процессы/планировщики:
  - там, где это допустимо правилами пентеста.

**Blue‑ответ**

- Усиленное журналирование:
  - PowerShell (Script Block, Module Logging).
  - WMI.
  - Sysmon.
- Детекты на поведение:
  - паттерны последовательности действий.
  - аномалии по времени/учёткам/хостам.
  - не только хеши/имена.
